ERRORES ENCONTRADOS DURANTE EL DESARROLLO Y SUS SOLUCIONES
====================================================

ERROR 1: serenity-bom version no disponible en Maven Central
------------------------------------------------------------
Mensaje: "Non-resolvable import POM: net.serenity-bdd:serenity-bom:pom:4.2.34"

Causa: El BOM (Bill of Materials) de Serenity solo existe en Maven Central
a partir de la version 5.3.7. La version 4.2.34 usada inicialmente no existe
como artefacto BOM en el repositorio central.

Solucion: Se eliminó el bloque <dependencyManagement> con el BOM y se
especificó la versión explícitamente en cada dependencia usando la version
4.0.46, que sí está disponible y verificada en Maven Central.

Lección: Antes de referenciar cualquier versión en un pom.xml, verificar
su existencia en https://repo.maven.apache.org/maven2/ consultando el
archivo maven-metadata.xml del artefacto.

------------------------------------------------------------
ERROR 2: Feature de Cucumber no parseado (# language: es)
------------------------------------------------------------
Mensaje: "FeatureParser Failed to parse resource... expected: #FeatureLine, got 'Feature:...'"

Causa: La directiva "# language: es" en el archivo .feature no fue
procesada correctamente por la versión de Gherkin incluida en
serenity-cucumber:4.0.46, dejando el parser en modo inglés e
interpretando "Feature:", "Scenario:", "Given", etc. como tokens
inválidos.

Solucion: Se reescribió el archivo .feature completamente en inglés
(Feature, Scenario, Given, When, Then, And) que es el idioma por defecto
de Cucumber y garantiza compatibilidad con todas las versiones del parser.
Las anotaciones del step definition se actualizaron de
"import io.cucumber.java.es.*" a "import io.cucumber.java.en.*".

Lección: Para evitar problemas de compatibilidad de idioma en Gherkin,
usar inglés como idioma base es la práctica más segura y portable.

------------------------------------------------------------
ERROR 3: Selector CSS interpretado como XPath por Serenity
------------------------------------------------------------
Mensaje: "Unable to locate element: {"method":"xpath","selector":"a.btn.btn-success.btn-lg"}"

Causa: El método findBy() de PageObject de Serenity, cuando recibe una
cadena de texto como "a.btn.btn-success.btn-lg", la interpreta
internamente como XPath en lugar de CSS selector. Un CSS selector como
"a.btn.btn-success" es inválido como XPath, por lo que el elemento nunca
se encuentra.

Solucion: Se reemplazó completamente el uso de findBy() y WebElementFacade
por WebDriverWait + By.cssSelector() / By.xpath() / By.id() de Selenium
puro en todos los Page Objects. Esto da control total sobre el tipo de
selector usado.

Adicionalmente, el botón "Add to cart" se localizó por texto visible con
XPath: //a[contains(text(),'Add to cart')]
en lugar de por clases CSS, lo que lo hace más robusto frente a cambios de
estilos en la UI.

Lección: En Serenity BDD, cuando se necesita precisión en el tipo de
selector, es más seguro usar WebDriverWait con By.* de Selenium directamente
que depender de la inferencia automática de findBy().

====================================================
HALLAZGOS TECNICOS DEL SITIO DEMOBLAZE
====================================================

1. ALERT NATIVO AL AGREGAR AL CARRITO
   DemoBlaze dispara un alert nativo del navegador (window.alert) al
   hacer click en "Add to cart". Se requiere driver.switchTo().alert().accept()
   para cerrar el popup y devolver el foco al driver. Si no se acepta,
   cualquier interacción posterior lanza una UnhandledAlertException.

2. CARGA ASINCRONA DEL CARRITO
   Los productos en el carrito se recuperan vía API REST de forma asíncrona.
   Pueden tardar entre 1 y 4 segundos en aparecer en la tabla #tbodyid.
   Se usó numberOfElementsToBeMoreThan() de ExpectedConditions combinado
   con una espera adicional (Thread.sleep) para garantizar que ambos
   ítems estén cargados antes de contarlos.

3. CONFIRMACION CON SWEETALERT
   La pantalla de confirmación final ("Thank you for your purchase!") usa
   la librería SweetAlert, que renderiza un modal HTML (no un alert nativo
   del navegador). El selector correcto es:
     - Título: .sweet-alert h2
     - Botón OK: .sweet-alert button.confirm
   Intentar manejar esta confirmación con driver.switchTo().alert() falla
   porque SweetAlert es un elemento del DOM, no un alert de JavaScript.

4. FORMULARIO PLACE ORDER CON BOOTSTRAP MODAL
   El botón "Place Order" abre un modal de Bootstrap (#orderModal).
   Es necesario esperar a que el modal sea visible antes de interactuar
   con los campos. El selector del botón de confirmación dentro del
   modal requiere contexto: //div[@id='orderModal']//button[contains(text(),'Purchase')]

5. SELECTOR DE PRODUCTOS EN CATÁLOGO
   Los enlaces de productos en la página principal usan la clase CSS
   "hrefch" (selector: a.hrefch). La búsqueda por texto con equalsIgnoreCase
   garantiza que el producto se encuentre independientemente del capitalizado.

====================================================
CONCLUSIONES GENERALES
====================================================

1. SERENITY BDD ES ADECUADO PARA PRUEBAS E2E
   La integración Serenity + Cucumber permite escribir pruebas legibles en
   formato BDD (Gherkin), con reportes HTML automáticos que incluyen
   capturas de pantalla por paso. Esto sirve como evidencia formal sin
   necesidad de grabaciones manuales.

2. EL PATRON PAGE OBJECT MEJORA EL MANTENIMIENTO
   Separar la lógica de cada pantalla (HomePage, ProductPage, CartPage)
   en clases independientes hace que los cambios en el UI solo requieran
   modificar una clase. Los step definitions permanecen estables.

3. LAS ESPERAS EXPLICITAS SON CRITICAS EN SITIOS ASINCRONOS
   DemoBlaze depende de APIs para cargar datos. Usar
   WebDriverWait + ExpectedConditions es indispensable para evitar
   NoSuchElementException y StaleElementReferenceException.
   Se recomienda evitar Thread.sleep() como estrategia principal.

4. VERIFICAR VERSIONES ANTES DE CONFIGURAR EL POM
   La compatibilidad de versiones en el ecosistema Maven es un punto
   crítico. Las versiones deben verificarse directamente en Maven Central
   antes de usarlas para evitar errores de resolución de dependencias.

5. USAR SELENIUM PURO CUANDO SEA NECESARIO
   Serenity BDD agrega abstracciones útiles, pero en algunos casos
   (como la inferencia de tipo de selector) puede generar comportamientos
   inesperados. Usar By.* de Selenium directamente con WebDriverWait es
   una alternativa más predecible y controlada.

6. MEJORAS POTENCIALES (fuera del alcance mínimo del ejercicio)
   - Implementar el patrón Screenplay (Actor/Task/Question) de Serenity
   - Parametrizar datos de prueba con archivo externo (JSON/CSV)
   - Agregar pruebas negativas: carrito vacío, formulario incompleto
   - Integrar con CI/CD (GitHub Actions)
   - Ejecución en modo headless para entornos sin interfaz gráfica
   - Agregar tags @smoke / @regression para diferentes suites de prueba
